# Morning Ben — MVP Design (A + Б-simple)

## Проблема

Утром, даже выспавшись, есть период тупняка: нет сил на душ/пробежку, но и просто
лежать не вариант — рука тянется к рилсам. Нужна альтернатива рилсам: короткие,
персонализированные, чуть более осмысленные «истории» из интересных фактов,
которые читаются в том же режиме тупого пролистывания, но оставляют что-то
взамен потраченного времени.

## Продукт (общее видение, не всё — MVP)

Telegram-бот. Заходишь, жмёшь «Дальше» — получаешь короткие сообщения-факты,
объединённые в одну историю (~15 сообщений на тему). Пролистал — следующая
история на другую тему. Истории готовятся заранее (ночью или непрерывно),
не генерируются на глазах. Дальше, вне MVP, это обрастает: реальными
источниками и картинками (Pinterest/YouTube), голосом персонажа (Бен Гросс),
памятью реакций и синком с Obsidian.

## Scope этой спеки

Только MVP = **компонент А (бот-доставка)** + **компонент Б-simple (генерация
из знаний модели, без внешних источников и картинок)**. Остальные эпики —
см. раздел «Бэклог» в конце, без детального дизайна.

## Архитектура

Два независимых процесса на одном VPS, связанных только через файловую систему —
никакого сетевого API между ними.

```
┌─────────────────┐         ┌──────────────┐         ┌─────────────────┐
│  Generator       │ пишет   │   stories/    │ читает  │  Bot             │
│  (Claude Code    │────────▶│   *.json      │────────▶│  (Python,        │
│   headless,      │         │               │         │   systemd,       │
│   cron)          │         └──────────────┘         │   long-polling)  │
└─────────────────┘                                    └─────────────────┘
        ▲                                                       │
        │ читает при генерации                                  │ пишет
        │                                                       ▼
┌─────────────────┐                                    ┌─────────────────┐
│ interests.md     │                                    │ state/           │
│ examples.md      │◀───────────────────────────────────│ user_state.json  │
│ (правишь руками) │  фидбек для антиповтора            │ shown_history.json│
└─────────────────┘                                    └─────────────────┘
```

- **Generator** — headless Claude Code (`claude -p ...`), запускается по cron.
  На вход: `interests.md`, `examples.md`, `shown_history.json` (чтобы не
  повторяться). На выход: несколько JSON-файлов в `stories/`.
- **Bot** — постоянно работающий Python-процесс (systemd), long polling
  Telegram API. Отдаёт истории из `stories/`, ведёт состояние пользователя
  и лог показанного.
- Оба компонента полностью независимы: генератор ничего не знает про Telegram,
  бот ничего не знает про то, как истории были сделаны.

## Форматы данных

**`interests.md`, `examples.md`** — обычный markdown, редактируется вручную
(SSH или позже через Obsidian-синк). Целиком идёт в промпт генератора без
парсинга структуры — просто текст.

**`stories/<id>.json`** — одна история:
```json
{
  "id": "2026-07-20-color-theory",
  "topic": "Цветовой круг и цветотипы",
  "generated_at": "2026-07-20",
  "messages": ["факт 1...", "факт 2...", "... до ~15"]
}
```
`messages` — массив текстов. Без картинок и видео (это R2).

`id` должен быть уникален при каждом запуске генератора (например,
включать timestamp, а не только дату) — иначе повторный запуск в течение
дня перезапишет уже показанные файлы вместо того, чтобы добавить новые.

**`state/user_state.json`** — где сейчас пользователь:
```json
{"current_story_id": "2026-07-20-color-theory", "current_message_index": 3}
```
Никакого `queue`-списка: следующая история каждый раз вычисляется на лету
(см. «Логика очереди» ниже).

**`state/shown_history.json`** — лог показанного, антиповтор + лёгкий фидбек:
```json
[
  {"id": "...", "topic": "...", "date": "2026-07-19", "messages_seen": 15, "completed": true},
  {"id": "...", "topic": "...", "date": "2026-07-19", "messages_seen": 2, "completed": false}
]
```
`completed: false` при малом `messages_seen` — сигнал «тема не зашла».
Генератор видит это в следующем промпте и не повторяет похожее. Задел под
будущую память (эпик D), без дополнительной работы сверх MVP.

## Логика очереди

«Очередь» — не статичный список, а живой запрос: **все файлы в `stories/`,
которых ещё нет в `shown_history.json`**. Дата в `id` — просто часть
идентификатора для уникальности файла, не используется для фильтрации.
Генерация может идти хоть раз в сутки, хоть непрерывно — бот не видит
разницы.

Момент, когда нужно определить «что дальше», наступает в двух случаях:
1. По `/start`.
2. Когда текущая история закончилась (после последнего «Дальше» или после
   «Следующая история»).

В обоих случаях бот пересканирует `stories/` на непоказанное. Если что-то
новое появилось (сгенерировалось, пока листала) — подхватывает и
продолжает. Если правда пусто — только тогда «Истории на сегодня
закончились, увидимся позже 🌙».

## Механика бота

- **`/start`** — всегда доступен, всегда пересобирает очередь заново
  («запустить ленту»), независимо от предыдущего прогресса. Показывает
  persistent-клавиатуру с двумя кнопками.
- **«Дальше ▶»** — присылает `messages[current_message_index]`, инкрементит
  индекс. Если это было последнее сообщение истории — сразу же после
  отправки шлёт короткий разделитель («✨ на сегодня всё по этой теме») и
  переключается на следующую историю (см. «Логика очереди»), индекс → 0.
- **«Следующая история ⏭»** — логирует текущую историю в `shown_history`
  с `completed: false` и `messages_seen = current_message_index`, шлёт тот
  же разделитель, переключается на следующую.
- **Очередь пуста** (после пересканирования) — «Истории на сегодня
  закончились, увидимся позже 🌙».
- **Один пользователь** — бот проверяет `chat_id` из конфига, всем
  остальным — тишина.

## Инфраструктура сервера

Чеклист того, что должно быть готово на VPS до первого запуска.

**Рантаймы:**
- Node.js (LTS) — для Claude Code CLI: `npm install -g @anthropic-ai/claude-code`
- Python 3.11+ и `venv` — для бота

**Авторизация Claude Code (разовая, руками):**
SSH на сервер → `claude` интерактивно один раз → он покажет ссылку логина →
открыть в любом браузере (не обязательно на сервере) → залогиниться Pro/Max
аккаунтом. Токен кешируется на диске сервера; дальнейшие headless-запуски
(`claude -p "..."`) переиспользуют его без интерактива.

**Headless-генератор:**
- Промпт — простой текстовый темплейт в репо (`generate_stories.md`),
  который на старте подставляет содержимое `interests.md`, `examples.md`,
  `shown_history.json` и просит записать N JSON-файлов в `stories/`.
- Non-interactive cron-запуск не может отвечать на permission-промпты →
  нужен `--dangerously-skip-permissions`, **строго в пределах рабочей
  директории проекта**, не на весь сервер.
- Скиллы (superpowers и т.п.) раннеру не нужны — это инструмент для
  разработки, а не для рантайма. Раннер — голый `claude -p` с готовым
  промптом.
- Cron: раз в ночь для старта (например 03:00). Частоту/непрерывность можно
  будет менять позже без изменений в боте — логика очереди это уже
  поддерживает.

**Бот:**
- systemd unit, автоперезапуск, long polling (без домена/TLS/вебхука)
- Токен от @BotFather — в `.env`, не в git
- `chat_id` владелицы — в конфиге

**Секреты:** только токен Telegram-бота + одноразовый браузерный логин
Claude Pro/Max. Никаких Pinterest/YouTube-ключей для MVP — это R2.

## Обработка ошибок и границы

Дизайн уже самортизирован тем, что очередь строится на лету из того, что
реально есть на диске — большинство сбоев вырождаются в «показываем
меньше», а не в падение:

- **Генератор упал / выдал невалидный JSON** — лог в `logs/generate.log`,
  файл не считается валидным. Бот его не увидит; в худшем случае очередь
  опустеет раньше — не крашится ничего.
- **Бот-процесс упал** — systemd поднимает заново; `user_state.json`
  пишется на диск после каждого нажатия кнопки, при рестарте `/start`
  восстанавливает ровно то место, где остановились.
- **Битый файл истории** (нет `messages`, кривой JSON) — пропускается и
  логируется при сканировании `stories/`, не падает боту в лицо.
- **Одновременные нажатия кнопок** — при одном пользователе и
  последовательной обработке апдейтов в `python-telegram-bot` не проблема
  для MVP, отдельно не обрабатывается.

## Тестирование (MVP-уровень)

Без автотестов на старте — личный проект на одного пользователя, важнее
скорость итерации:
- Ручной прогон генератора из терминала — глазами проверить валидность
  JSON и соответствие стилю `examples.md`.
- Ручной прогон бота с тестовым `chat_id` — пройти весь цикл: `/start` →
  несколько «Дальше» → «Следующая история» → дойти до конца очереди →
  добавить новый файл в `stories/` руками → убедиться, что «закончились»
  не показалось раньше времени.
- Валидация схемы при чтении файла в боте — основная защита от мусора,
  тестов сверху не нужно.
